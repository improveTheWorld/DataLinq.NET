# /update - Local Context Update

This workflow **appends new information** to the existing `.agent/context/` files based on the current conversation context. Unlike `/scan` which regenerates everything, `/update` is **local and targeted** — it only adds what's relevant to what you're currently working on.

## When to Use

Invoke `/update` when:
- You've made changes to the codebase that should be documented
- You discovered new components, patterns, or documentation
- The current context is missing information you just encountered
- You want to persist knowledge gained during this session

## Key Principle

> **Additive, not destructive.** Append new information; don't overwrite existing content.

## Execution Steps

### Step 1: Identify the Scope

Based on the current conversation, determine what type of information needs to be added:

| If you learned about... | Update this file |
|-------------------------|------------------|
| New patterns, design decisions | `context/architecture.md` |
| New/changed source folders | `context/source.md` |
| New documentation files | `context/docs.md` |
| New test projects or commands | `context/tests.md` |
| New build/pack/release info | `context/packaging.md` |
| Project-specific AI guidance | `guidance.md` (create if needed) |

### Step 2: Navigate to the Relevant Area

Explore the codebase locally around the area of interest:
- List directories that were mentioned in conversation
- Check for new files that might not be indexed
- Read any new documentation or configuration

### Step 3: Read Existing Context

Read the target context file(s) to understand:
- What's already documented
- The existing format and style
- Where new info should be inserted

### Step 4: Append New Information

Add the new information **in the same style** as existing content:
- Use matching table formats if adding to a table
- Use matching list styles if adding to a list
- Add to the appropriate section (don't create duplicates)

### Step 5: Update Timestamp

Add an update note to `index.md`:
```markdown
> Auto-generated by `/scan` on [ORIGINAL], updated by `/update` on [TODAY]
```

### Step 6: Confirm Changes

Tell the user:
- What file(s) were updated
- What information was added
- Any observations about gaps still remaining

## Scope Boundary

**DO:**
- Add information directly related to current conversation
- Update sections that are clearly outdated
- Create `guidance.md` if user provides project-specific instructions

**DON'T:**
- Regenerate entire files (use `/scan` for that)
- Add speculative information not confirmed in conversation
- Remove existing information unless explicitly wrong

---

## Context Maintenance

### Resolve Contradictions
If you find contradictory information between conversation and indexed context:
1. Investigate to determine which is correct
2. Update the context file to reflect accurate information
3. Clarify ambiguous statements to prevent future confusion
4. Notify user: "Resolved contradiction in [file]: [old] → [new]"

### Manage File Size (Hierarchy)
If adding information would make a context file too large (>200 lines):
1. Keep a **general summary** in the original file
2. Create a **sub-file** for the new detailed topic
3. Add a link from the parent file to the sub-file
4. Update `index.md` if the structure changed

**Example:**
```
Adding detailed extension docs to source.md (already 180 lines)
  ↓ Instead:
1. Keep source.md as overview
2. Create context/source/extensions.md with details
3. Add link in source.md: "For details, see [extensions.md](source/extensions.md)"
```

---

## Notes

- `/update` is **local** — focused on current work context
- `/scan` is **global** — regenerates all context files
- `/refresh` is **read-first** — reads context, may enrich if gaps found
- Always preserve the existing format and style of context files
