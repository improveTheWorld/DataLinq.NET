# /scan - Repository Context Index Generator

This workflow creates a structured index of the repository that helps AI assistants understand the codebase quickly.

## Output Structure

After running this workflow, the following structure will be created:

```
.agent/
├── index.md              # Entry point: project overview + navigation
└── context/
    ├── architecture.md   # Core patterns, design decisions
    ├── source.md         # Source code organization
    ├── docs.md           # Documentation structure
    ├── tests.md          # Test organization & commands
    └── packaging.md      # Build, pack, release info
```

---

## Execution Steps

### Step 1: Discover Repository Structure

1. List the root directory to identify top-level organization
2. Identify key folders by category:
   - **Source**: folders with `.csproj`, `.cs`, `src/`, etc.
   - **Docs**: folders with `.md` files, `docs/`, `documentation/`
   - **Tests**: folders with `test`, `tests`, `*Tests` in name
   - **Packaging**: `.nuspec`, `nupkg/`, `packages/`, `publish` scripts
   - **Config**: `.github/`, `.vscode/`, `*.props`, `*.targets`
3. Note the main solution file (`.sln`) location

### Step 2: Create index.md

Create `.agent/index.md` with:

```markdown
# [Project Name] - AI Context Index

> Auto-generated by `/scan` on [YYYY-MM-DD]

## Quick Summary

[Write 1-2 paragraphs describing:
- What this project does
- Primary technology/language
- Key architectural patterns (if identifiable from README)]

## Repository Structure

[ASCII tree showing key folders, max 2-3 levels deep]

## Navigation Guide

| Task | Start Here |
|------|------------|
| Understanding architecture | [context/architecture.md](context/architecture.md) |
| Finding source code | [context/source.md](context/source.md) |
| Working with documentation | [context/docs.md](context/docs.md) |
| Running tests | [context/tests.md](context/tests.md) |
| Building/releasing | [context/packaging.md](context/packaging.md) |

## Quick Commands

[List common commands discovered: build, test, run, pack]
```

### Step 3: Create context/architecture.md

Analyze README and key docs to extract:
- Core design patterns mentioned
- Layer/component organization
- Key abstractions and their relationships
- Technology stack

### Step 4: Create context/source.md

List source folders with brief descriptions:
- Component name → folder path → 1-line purpose
- Group by layer if applicable (Framework, Extensions, Data, etc.)

### Step 5: Create context/docs.md

Catalog all documentation files:
- File name → 1-line summary of content
- Group by topic if many files exist

### Step 6: Create context/tests.md

Document:
- Test project locations
- How to run all tests: `dotnet test` or equivalent
- How to run specific test projects
- Any special test categories or fixtures

### Step 7: Create context/packaging.md

Document:
- Build commands
- Pack commands (NuGet, npm, etc.)
- Publish/release commands
- Version management (where version is defined)
- CI/CD references if present

---

## Update Mode

If `.agent/index.md` already exists:
1. Ask user: "Index exists. Update or regenerate from scratch?"
2. **Update**: Only refresh sections that have changed
3. **Regenerate**: Delete and recreate all context files

---

## Context Maintenance

### Resolve Contradictions
If you find contradictory information while scanning:
1. Investigate to determine which is correct
2. Document only the accurate information
3. Use clear, unambiguous language to prevent confusion

### Manage File Size (Hierarchy)
If a context file would become too large (>200 lines):
1. Keep a **general summary** in the main file
2. Create **sub-files** for detailed topics (e.g., `context/source/extensions.md`)
3. Add links from the parent file to the sub-files
4. Update `index.md` to reflect the new structure

**Example structure:**
```
context/
├── source.md              # Overview + links to sub-files
└── source/
    ├── data-layer.md      # Detailed: Data.Read, Data.Write
    ├── extensions.md      # Detailed: All extension projects
    └── framework.md       # Detailed: Framework components
```
